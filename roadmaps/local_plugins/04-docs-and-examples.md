# Docs and Examples

## Goal

Document directory plugin sources clearly for local development, Docker use, and
plugin author workflows.

## Background from the discussion

This feature is mainly for development. It should help developers build a small
mock plugin and test executor changes without creating or committing a git
repository. It still needs to work with Docker deployments when paths are bind
mounted consistently.

## Scope

- Update user-facing docs that describe plugin repository sources.
- Update any generated or static reference text that lists supported source
  forms.
- Add Docker bind mount guidance.

## Out of scope

- Do not present `dir://` as the preferred production deployment mechanism.
- Do not remove guidance for GitHub or local git repos.
- Do not add a broad tutorial unless docs already have a natural place for it.

## Required behavior

- Docs list all supported source forms:
  - GitHub shorthand
  - GitHub shorthand with ref
  - explicit GitHub URL
  - explicit GitHub URL with ref
  - `file:///absolute/path/to/repo`
  - `dir:///absolute/path/to/plugin`
- Docs explain the difference between `file://` and `dir://`.
- Docs state that raw filesystem paths remain unsupported.
- Docs state that `dir://` does not support refs.
- Docs state that `dir://` copies a snapshot on refresh or pull and includes
  uncommitted edits.
- Docs state that workers still need refresh/restart to see changes.
- Docker docs explain that the same absolute `dir://` path must be reachable in
  API and worker containers, for example:

```yaml
volumes:
  - ./mock-plugin:/plugins/mock-plugin
```

```json
{"source": "dir:///plugins/mock-plugin"}
```

## Implementation notes

- Preserve the existing language that `file://` local repositories are
  committed-code sync sources.
- Use "development directory source" or similar wording to avoid production
  ambiguity.
- Keep examples short and operational.
- If `docs/public/llms.txt` is generated by a docs script, regenerate it rather
  than hand-editing it. If not, update it directly.

## Tests and verification

- Build docs if Node dependencies are available:

```bash
npm run build --prefix docs
```

- At minimum, search for stale statements that say only GitHub and `file://`
  sources are supported:

```bash
rg -n "file://|GitHub entries|local git repositories|plugin repositories" docs README.md
```

## Completion criteria

- All source lists include `dir://` where appropriate.
- The development-only and Docker path requirements are clear.
- No docs imply that raw paths are supported.
- No docs imply that `file://` includes uncommitted changes.
